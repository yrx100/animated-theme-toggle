# 商业净室重建设计

## 目标

为 `project-refactor` 增加“商业净室重建（Commercial Clean Room Rebuild）”模式。该模式用于在保留旧项目用户可见功能与接口兼容的前提下，独立重建新项目，并将商业发布所需的权利、供应链、品牌和质量风险设为可验证的发布闸门。

该模式提供工程与许可证风险控制，不提供法律意见，也不得输出“绝对可以商用”的保证。

## 触发与证据等级

用户明确要求“标准净室”“商业净室”“Clean Room”或等效表达时启用。模式应在每次交付中声明证据等级：

- **工程净室**：同一实施主体曾阅读旧源码；以行为规格、独立设计和相似性审查控制复制风险，不能声称严格法律净室。
- **角色隔离净室**：旧项目分析者只交付行为规格，实施者只读取该规格；保留输入范围、交接物和审查记录。

## 前置商业预检（硬门槛）

在普通只读盘点之前执行。生成《商业净室预检报告》，覆盖：

1. 旧项目、直接与传递依赖、插件、容器基础镜像、代码生成器、模型和数据集的来源、版本、许可证、版权声明、NOTICE、贡献者协议及附加条款。
2. Logo、图标、图片、字体、截图、文案、示例数据、音视频、项目名、包名、域名、产品描述、作者信息和遥测端点。
3. 旧项目中需保持兼容的公开行为、接口、配置、数据格式、错误语义和性能目标。

非商用、不得衍生、许可证或权属未知、品牌/素材未获授权、来源不可确认时，一律标记为“商业发布阻断”。未解除阻断项时，不得进入重建实施，也不得作出可商用结论。可继续提供替换、剔除或人工审核建议。

## 规格、设计与实施

预检通过后，先把允许信息转化为版本化的《兼容性规格》和《功能验收矩阵》。矩阵逐项记录输入、输出、边界条件、异常、性能目标、兼容状态和自动化验收证据；没有证据的功能不得标记完成。

实施者不得直接复用旧项目的代码、目录及模块布局、符号和文件命名、注释、测试、文案、资源、默认配置或私有协议实现。先制定独立架构、模块边界和命名方案，再开始实现。旧项目只能作为行为与接口的受控来源。

## 发布闸门

发布前必须完成以下复检：

1. 功能验收矩阵全部通过，或每个不兼容项都经用户明确接受。
2. 相似性扫描覆盖源码、文件结构、公共符号、注释、文案、配置、测试、资源、构建产物和 Git 历史。必要相似项须记录原因。
3. 所有品牌、项目标识、素材与内容替换为独立且来源可证的版本。
4. 生成并审查 SBOM（推荐 SPDX），核验全部依赖及供应链组件的许可证、来源和已知漏洞。
5. 执行安全和发布检查：密钥泄露、依赖漏洞、构建完整性、日志/数据脱敏、第三方服务条款、备份恢复与漏洞响应要求（按项目类型裁剪）。

最终只允许输出三档状态：`允许发布`、`条件允许`、`阻断发布`。每一项状态必须附上证据、未决风险、解除条件和责任主体；许可证、商标、专利、隐私等法律问题必须保留“需法务审核”状态，不能由 Skill 自动放行。

## 长任务持久化与恢复

执行契约确认后、修改业务代码前，创建 `specs/NNN-功能名/` 过程目录，保存规格、计划、调研、任务、阶段账本、商业闸门与交接状态。每次恢复先读取这些文件并核对 Git 状态与基线；每个可验证阶段完成后立即更新账本和交接状态。

根 `AGENTS.md` 与 `CLAUDE.md` 保存内容相同、由 `SPECKIT START/END` 包围的最小引导块：当前活跃 feature、必读过程文件、硬边界、上次验证状态与恢复规则。已有入口文件只能修改该边界区块，不能覆盖既有项目指令。入口文件不存放完整技术细节，详情保留在对应 `specs/NNN-功能名/` 内。

## 交付物

- 商业净室预检报告
- 行为兼容性规格与功能验收矩阵
- 独立架构与命名记录
- 组件/资产来源清单与 SPDX SBOM
- 相似性审查与替换记录
- 发布闸门报告：状态、验证命令及结果、阻断项、人工动作和回滚方案

## 完成判定

只有当范围内功能的验收证据完整、商业发布阻断项均已解除、所有资产与依赖都有可接受的权利来源、发布复检通过时，项目才能标记为“允许发布”。该结论是工程交付状态，不替代适用法域下的律师意见。
