# 项目重构

将重构视为保留或明确迁移行为的工程工作，而不是格式化或大规模替换。默认先分析，不修改代码。

用户要求“标准净室”“Clean Room”“商业净室”“独立重建”或等效表达时，启用下文的**商业净室重建模式**。该模式以可审计的工程、许可证和供应链检查降低商用风险；它不是法律意见，不能承诺“绝对可商用”，也不能取代适用法域下的书面授权或律师审核。

## 工作原则

- 先理解项目、基线和用户目标，再提出方案。
- 以最小可验证阶段推进；每阶段都可检查、回滚和独立交付。
- 保持公开 API、数据、配置和运行时语义；若要改变，必须在执行契约中明确兼容、迁移和回滚计划。
- 不读取、展示、提交或上传密钥、令牌、`.env` 内容和客户数据。默认排除依赖目录、构建产物、生成代码、vendor 和二进制文件。
- 技术选型或升级时，只查官方文档核验维护状态、兼容性和迁移要求。
- 不把测试通过、代码重写或开源仓库可访问性误报为拥有商用权利；许可证、商标、专利、隐私或资产权属存在不确定性时，明确标记“需人工/法务审核”。

## 商业净室重建模式

### 适用边界与证据等级

此模式默认目标是保持旧项目的用户可见功能、公开接口、配置和数据格式兼容，同时独立设计和实现新项目。旧项目只能是受控的需求与可观察行为来源，不能作为可复用实现来源。

在每次报告中声明以下证据等级之一：

- **工程净室**：同一实施主体曾访问旧项目源码。通过行为规格、独立设计和相似性审查降低复制风险；不得宣称严格法律净室。
- **角色隔离净室**：旧项目分析者只交付经审查的行为规格；实施者只读取该规格，不读取旧项目源码、测试、资源或内部设计。记录角色、允许输入和交接物。

无论等级如何，禁止复制、改写或迁移旧项目的代码、目录/模块布局、文件与符号命名、注释、测试、文案、Logo、图标、图片、字体、音视频、截图、示例数据、默认配置、版权头、作者信息、项目名、包名、域名、遥测端点和私有协议实现。功能相同所必需的相似接口须在相似性审查中单独说明。

### 前置商业预检（硬门槛）

在“阶段 1：只读盘点与基线”之前执行，且只做只读检查。创建《商业净室预检报告》，逐项记录来源、版本、证据、风险、处理建议和状态：

1. 旧项目的 `LICENSE`、`NOTICE`、版权声明、贡献者协议、商标/品牌声明和其他附加条款。
2. 直接与传递依赖、插件、容器基础镜像、代码生成器、模型、数据集和第三方服务的许可证、来源、商用限制与归属。
3. 所有品牌与内容资产：项目名、Logo、图标、字体、图片、文案、截图、示例数据、音视频、域名和包名。
4. 需兼容的用户可见功能、公开 API/命令、配置、数据格式、错误语义和性能目标；仅记录行为规格，不摘录实现表达。

发现以下任一项时，状态为**商业发布阻断**：非商用或不得衍生限制、许可证/权属/来源不明、许可证冲突、未获授权的品牌或内容资产，或无法确认的第三方服务条款。阻断时不得开始重建实施、不得声称可商用；可继续给出剔除、独立替代、书面授权或法务审核的解除建议。用户必须在执行契约中提供书面授权、人工审核结论或明确排除受阻范围，才能解除相应阻断项。

通过预检后，先形成版本化《兼容性规格》和《功能验收矩阵》。矩阵对每项功能记录输入、输出、边界条件、异常、性能目标、兼容状态、证据来源和自动化验收；没有验收证据的功能不得标记为已完成或已兼容。

## 阶段 1：只读盘点与基线

除非用户已明确授权实施，否则只做只读检查。识别：

1. 语言、框架、包管理器、构建/测试/lint/typecheck 命令和运行入口。
2. Git 仓库、当前分支、未提交改动、工作树状态和 monorepo 边界。
3. 架构边界、依赖方向、循环依赖、重复实现、过大模块、耦合热点、错误处理与测试薄弱点。
4. 现有质量基线：测试、构建、类型检查、lint 及其失败项。区分原有失败和本次引入的失败。

商业净室重建模式下，盘点还必须将《商业净室预检报告》《兼容性规格》和《功能验收矩阵》纳入基线；旧项目的源码、测试、资源和内部架构不得成为新项目设计或实现的输入。

没有 Git 仓库、工作树不干净、无法确定项目根目录或发现密钥风险时，不自行初始化、清理、暂存或覆盖。说明限制，并要求用户把相应授权写入最终执行契约。

## 阶段 2：输出决策简报

用户未指定目标时，使用简洁中文输出：

- 项目概览与当前基线；
- 最重要的重构机会、证据和风险；
- 2–3 套方案，分别给出目标、关键改动、受影响模块、收益、代价、兼容性风险和验证方式；
- 推荐方案及理由；
- 依赖新增、替换或升级及官方依据；
- 建议的阶段划分和预估的高风险操作。

商业净室重建模式下，还必须输出：证据等级、预检结论与每个商业发布阻断项、允许和禁止的旧项目输入、独立架构/命名方向、功能验收矩阵覆盖状态、资产与依赖替换方案，以及最终发布闸门的预期证据。没有解除所有阻断项的路径时，明确说明不能进入实施。

不要在此阶段改文件。用户可选择方案、缩小范围、要求展开或回复“按推荐方案执行”。

## 阶段 3：一次性执行契约

在开始任何写入前，汇总为一份可直接确认的执行契约。它必须包含：

- 选定方案、目标与非目标；
- 阶段、模块/文件范围、依赖与版本变更；
- 预计删除或覆盖的文件；
- Git 操作（建分支、提交、推送、历史改写）；
- 数据库迁移/清理、部署/发布、生产访问、第三方上传等外部高风险操作；
- 将运行的构建、测试、安装、lint、typecheck 与集成验证命令；
- API、数据库、配置或行为变化的兼容、迁移与回滚计划；
- 已知限制、资源上限和完成判定。

商业净室重建模式下，契约还必须包含：

- 证据等级和角色隔离安排；旧项目允许查看的范围、禁止输入及交接物；
- 《商业净室预检报告》的每个阻断项及其书面授权、人工审核、剔除或替换的解除条件；
- 《兼容性规格》《功能验收矩阵》、独立架构和命名方案；
- 依赖/资产来源清单、SBOM 格式与审查方式；
- 相似性扫描范围（源码、结构、符号、注释、文案、配置、测试、资源、构建产物和 Git 历史）；
- 发布闸门、安全检查、隐私/数据处理与第三方服务条款的适用范围；
- 最终状态只能为“允许发布”“条件允许”或“商业发布阻断”，以及每种状态的完成条件。

用户确认该契约一次后，进入自主执行模式：不再因阶段、文件修改、验证、Git 操作或已列出的高风险操作向用户二次确认。

未在契约中列出的新高风险操作一律跳过，继续完成其余安全范围内工作，并将跳过原因和人工后续项写入最终报告。不要停下来等待用户。

## 阶段 4：自主实施

1. 若契约允许且仓库状态满足条件，创建 `codex/refactor-<topic>` 分支；否则在契约指定的工作区执行。
2. 根据风险、模块边界、耦合度和验证成本自动拆分阶段，不用固定文件数或行数机械判断。
3. 每阶段先用特征测试或现有测试锁定关键行为，再重构内部结构。优先小步、可逆变更，避免混入无关格式化。商业净室重建模式下，仅能从《兼容性规格》和《功能验收矩阵》导出测试；不得翻译、复制或改写旧项目测试。
4. 在契约授权范围内自行运行项目建议的完整验证流程，并修复本次改动造成的失败。对缺失的关键回归覆盖，补充测试。
5. 对每阶段记录：实际改动、依赖变化、命令及结果、与基线差异、未解决风险、回滚点和下一阶段状态。不中途等待确认。
6. 达到资源上限、遇到无法解决的外部依赖或未授权高风险操作时，保留已验证成果，跳过受阻部分并继续可完成的部分。

商业净室重建模式下，实施前先完成独立架构、模块边界、文件布局和命名设计。实施中持续记录功能规格来源、独立设计理由、替换资产/依赖的来源与许可证。发布前运行相似性审查，覆盖源码、文件结构、公开符号、注释、文案、配置、测试、资源、构建产物和 Git 历史；不必要的相似项必须删除或独立重写，必要相似项必须记录功能或行业惯例依据。

发布前还必须生成并审查 SBOM（优先 SPDX），检查依赖许可证、来源、已知漏洞和构建完整性；按项目范围检查密钥泄露、日志/数据脱敏、第三方服务条款、备份恢复与漏洞响应要求。检查失败或证据缺失时，保留“商业发布阻断”状态。

## 完成与交付

完成后直接给出中文交付报告：

- 重构目标和实际完成的阶段；
- 改动模块/文件与关键设计取舍；
- 新增、移除或升级的依赖；
- 运行过的验证命令、结果，以及相对基线的变化；
- 新增的回归测试；
- 未完成/跳过项、原因、风险和建议的人工动作；
- 分支、提交与回滚方式。

商业净室重建模式下，额外交付：

- 商业净室预检报告、证据等级与角色/输入隔离记录；
- 兼容性规格、功能验收矩阵及每项通过或不兼容的证据；
- 独立架构与命名记录、组件/资产来源清单、SBOM；
- 相似性审查结果、替换记录、安全与供应链检查结果；
- 发布闸门状态（仅限“允许发布”“条件允许”“商业发布阻断”）、未决风险、解除条件和需人工/法务处理项。

明确区分“已验证”“基线已失败”“本次失败”“未验证”。商业净室重建模式还必须明确区分“工程检查通过”与“已获法律/权利授权”，不得将前者表述为后者。除非契约明确要求，不自动把分支推送到远端或部署。
